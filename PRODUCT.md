# Product

## Register

brand

## Users

Lightning P2P serves Windows users who need to move large files without accounts, upload caps, or temporary cloud links. Primary users include developers moving builds and databases, creators moving raw media, open-source evaluators comparing transfer tools, and early mobile beta testers who want laptop-to-phone transfer without a browser-only compromise.

They usually arrive with a practical job: send a file now, understand whether the transfer is direct or relayed, and trust that the bytes are private and verified. The website also serves technical skeptics who need proof before accepting speed, privacy, or security claims.

## Product Purpose

Lightning P2P is a free, open-source peer-to-peer file transfer app built with Rust, Tauri, iroh, and iroh-blobs. It exists to make large file transfer direct, encrypted, verified, and simple enough for everyday use while staying honest about benchmark evidence.

Success means users can install the app, send huge files directly when networks allow it, fall back through relay when needed, understand the route, and see published methodology before any strongest speed claim.

## Brand Personality

Credible, technical, and fast. The voice is lab-grade without being cold: specific about routes, hashes, release artifacts, and benchmarks, but direct enough for non-experts to understand.

The brand should feel like a speed lab workbench: measured, bright at the point of proof, and intentionally anti-hype.

## Anti-references

- Generic SaaS card stacks with vague productivity copy.
- Purple or purple-blue gradient landing pages.
- Absolute "fastest" claims without published benchmark evidence.
- Cloud-storage marketing language that hides where files go.
- Mobile web transfer promises that bypass the native iroh path.
- Decorative glassmorphism, side-stripe callouts, gradient text, and repeated icon cards that do not add proof.

## Design Principles

1. Proof before claim. Speed, privacy, and reliability copy must attach to route data, benchmarks, release artifacts, or source code.
2. Show the route. Direct, relay, and mixed routing should be visible in app UI and explained plainly on the website.
3. Make multi-device help explicit. Swarm or multi-provider acceleration is opt-in, scoped to active shares, and described as a measured capability, not magic.
4. Native phones, not weakened phones. Android and iOS are native targets so transfer logic stays Rust plus iroh plus iroh-blobs.
5. Technical does not mean obscure. The best copy gives users enough detail to trust the system without asking them to read the source first.

## Accessibility & Inclusion

Target WCAG AA contrast for all public website and app surfaces. Route status must never depend on color alone. Interactive controls should meet 44 px touch targets on phone widths. Motion should be restrained, respect reduced motion preferences, and never block transfer actions. Text must remain readable at 390 px, 768 px, and desktop widths without overlap.
